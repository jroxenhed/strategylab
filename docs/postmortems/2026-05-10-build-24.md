## What I followed well

- **Single-cycle pipeline.** Picked 3 tasks (1 P1 each + 1 [next]) with verified file-independence: F91 lives in `backtest_quick.py`, F94 spans `data.py`/`indicators.py`/`backtest.py`, F86 is brand-new `middleware.py` + `main.py` wiring. Zero overlap meant the original implementation pass had no merge friction.
- **Round 2 dispatch was correctly scoped.** P1 findings (3 smuggling vectors + 1 hang) triggered a re-review on the changed lines, not on the whole diff again. Used correctness + adversarial (the original P1-flagging persona), per §4 fix-loop step 3. Round 2 returned clean (0 P0/P1, only 2 P3 advisories) — exactly the signal §4 expects for a successful fix loop.
- **Cross-reviewer convergence drove confidence.** Both `replay-receive double-call hang` (reliability P2 + adversarial P2) and `source-check duplication + case sensitivity` (kieran P2 + adversarial P2) had two independent reviewers flag the same finding. Treating those as high-signal-apply and the single-reviewer P2s as judgment calls produced a tight, well-justified fix set.
- **Helper extraction at the inflection point.** 3 routes already had the F94 source check duplicated; refactoring all 5 sites (3 new + 2 existing in `quote.py`) to a shared `require_valid_source(source)` helper was DRY done at the right time — not premature, not after the pattern decayed.
- **Adversarial-hardened middleware in-PR.** The smuggling vectors (duplicate CL, TE+CL coexistence, CL=0+chunked) would all have been P0s if shipped to production. Catching and fixing them inside the same PR via the adversarial reviewer is exactly what the multi-agent review is for.

## What I skipped or substituted, and why

- **Substituted dedicated reviewer agent types with `general-purpose`.** Every `compound-engineering:review:*-reviewer` agent name (correctness, testing, adversarial, security, kieran-python, reliability) returned `Agent type ... not found`. This is the 4th consecutive overnight (builds 21/22/23/24) hitting this gap. F80 already codifies that the routine env doesn't load the plugin registry. To preserve persona framing, I prefixed each prompt with an explicit "You are the X REVIEWER persona. Stay in your lane." block. This recovers most of the behavioral specialization but not whatever system-prompt scaffolding the dedicated agents carry.
- **Skipped the full `uvicorn` smoke test.** `backend/venv/` is still missing in the routine container — same as builds 21–23. Substituted with AST + import-time check + a standalone `asyncio.run`-driven middleware behavior smoke (duplicate-CL rejection, TE+CL slow-path, replay-empty contract). The substitute is what §3.5 documents as the fallback; F97 tracks the real fix.
- **Did not apply round-1 P2 finding "F104 — empty-list contract drift between F69 and F91"** (reliability 0.88). The fix would touch `WatchlistRequest._validate_symbols` which is outside the F91 scope and would expand the test surface. Filed as F104 for the next run.
- **Did not apply security P2 0.85 — `trailing_stop: Optional[dict]` → `Optional[TrailingStopConfig]`** in `QuickBacktestRequest`/`BatchQuickBacktestRequest`. Same scope-creep reasoning: untouched by F91 and would need new tests for the typed-model coercion. Filed as F103.

## Friction points

- **Routine env F80 gap.** The dedicated reviewer agents have failed to resolve in every overnight build since 21. This is no longer "the new normal will probably go away" — it's a persistent infrastructure gap that's costing the build measurable specialization signal. The persona-prompt-injection workaround is reasonable but not free. **The codified F80 manual-dispatch fallback should be the documented happy path for the routine env, not a fallback** — the prompt patch's protocol step says "use the Task tool with the explicit `subagent_type` shown" which is impossible here.
- **6 reviewers in parallel ≠ 6× speed.** Wall-clock dominated by the slowest reviewer (testing took ~4 minutes; the median was ~90 seconds). Future protocol: dispatch the conditional personas (kieran-python, reliability) on their own slightly-delayed wave so the always-on roster doesn't wait for them. Or use sonnet for the always-on + opus for the longest-running.
- **Pydantic field-validator ordering surfaced as a defense-in-depth gap (adversarial P2 0.72).** Reviewer noted that `direction` validator fires before `symbols`, so a malformed direction short-circuits the symbol count check. Net impact is zero (the request is rejected either way) but a Pydantic refactor could expose this. Documenting in CLAUDE.md or the prompt patch would prevent the trap.
- **`get_available_providers()` returns `list(_providers.keys())`** — a fresh snapshot per call. F94's check happens at the route boundary and `_fetch` re-reads `_providers.get(source)` later. If a provider is unregistered between the two reads (e.g. IBKR disconnect mid-request), the error shape differs ("Invalid source" → "Unknown data source"). Reliability P3, not actioned. Worth a CLAUDE.md note about the provider-registration lifecycle.

## Recommendations

- **Document F80 manual-dispatch as the routine-env happy path.** `docs/overnight-builder-prompt-patch.md` §4 currently presents `subagent_type: compound-engineering:review:correctness-reviewer` as the canonical path. After 4 builds of failure, flip the doc: present manual `general-purpose` + persona-prompt-injection as the canonical routine-env approach, with the dedicated agents as "available in interactive sessions / future routine env upgrades". Removes the dispatch ceremony.
- **File the dedicated-agent install issue against the routine builder container.** This is the root cause of the F80 gap. Add to NEXT_RUN ops backlog.
- **Add a CLAUDE.md note about provider-lifecycle assumptions.** The `_providers` dict is populated once at lifespan startup (init_ibkr) and effectively read-only after. F94's TOCTOU is benign because of this invariant, but a future refactor that adds dynamic re-registration could regress it silently.
- **Pre-allocate the F102–F113 follow-ups in TODO.** The new entries are sorted into buckets (hardening/polish/testing/arch) but tomorrow's run has to decide which to pick. Tagging F102 [next] is good; F104 (empty-list contract harmonize) should arguably also be [next] since it's the smallest unblock with the cleanest scope.
