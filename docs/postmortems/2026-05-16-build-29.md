## What I followed well

- **F136 Tier A protocol applied correctly.** F220 fits all three Tier A criteria: bundle of `[easy]` items, total diff <100 lines (34), no contract surface. No personas dispatched. Orchestrator verification covered: `npm run build` clean, `vitest run` 238/238 pass (unchanged from baseline — so the migration didn't break any of the existing chart-mount, MiniSparkline, Results, PerformanceComparison-adjacent tests), `bin/preview-smoke.sh` pass, and grep verification confirmed zero `new ResizeObserver` callsites remain.
- **Spot-checked file independence before editing.** The three sites are in three separate features (`trading/MiniSparkline.tsx`, `discovery/PerformanceComparison.tsx`, `strategy/Results.tsx`) with no shared logic — applied edits sequentially since this is a serial Opus run, not a parallel implementer dispatch.
- **F218 trap awareness.** When dropping the ResizeObserver in MiniSparkline, I recognised that the `applyRange()` helper still calls `fitContent()` on data updates (line 164), so the resize-time `fitContent()` loss is only visible if the user resizes between data ticks. Documented this as F250 instead of re-adding an RO — adding an RO that only calls `fitContent()` (not `applyOptions`) would be safe, but it's only worth doing if users actually notice the gap.
- **PerformanceComparison.tsx container-height move.** Spotted that the chart's `height: 200` was load-bearing because the container div had no explicit height (`style={{ width: '100%', borderRadius: 4, overflow: 'hidden' }}`). Moved `height: 200` to the container so `autoSize: true` has a non-zero height to observe.

## Friction points

- **Tier A's "matching review depth to risk" still requires careful self-classification.** The contract-surface override list in CLAUDE.md is binary (Pydantic / error wording / auth / persistence / TS shared types / public route removed). UI behaviour changes that affect rendering correctness — e.g. "the chart still resizes correctly when its container changes" — aren't on that list, but they ARE user-visible. Tier A says "rendering conditions, hover/popover, click handlers, timers, AbortControllers, focus listeners" trigger Live-Browser verification; the autoSize migration arguably falls under "rendering conditions" since it changes how the chart canvases resize. Without a browser, the static-grep + tests + preview gate is the best available substitute, but the protocol's "skippable" exit clause ("the test path needs unrealistic setup") isn't quite a fit either. Honest answer: F220 should have been browser-verified in a session that had `chrome-devtools-mcp`, but doing so here would have failed cleanly without the MCP. The protocol could codify "if the MCP is unavailable, fall back to static verification + flag in PR" as an explicit Tier A path rather than treating it as an implicit substitution.
- **chrome-devtools-mcp gap is recurring.** This is the third F-item I've seen (F210, F212, now F220) where the protocol's `[next]` tag implies it's pickable but the verification step requires an MCP that isn't loaded in the routine env. The protocol's existing answer is "either via paper-trading session or via headless saved-strategy seeding" — both manual interactive paths — but the overnight builder has neither. Worth tagging items that *require* live-browser explicitly so they don't bubble up to `[next]` in the routine env's planning view.
- **F-item ID allocation.** Had to grep for the highest used F-ID (F248) to allocate F249 + F250. The pre-commit hook stamps dates and runs `sync-todo-index.py`, but it doesn't allocate IDs. A minor friction; could be addressed by a `bin/next-fid.py` helper.

## Recommendations

- **Codify the static-fallback Tier A path for Live-Browser when MCP is unavailable.** Add a paragraph to CLAUDE.md "Live-Browser UI Verification" stating: "If the routine builder env lacks `chrome-devtools-mcp`, fall back to (a) build + tests + preview smoke, (b) static grep verifying the pattern matches a previously-browser-verified canonical site, (c) flag in PR description for morning reviewer. Note 'static-verified only — chrome-devtools-mcp unavailable' in JOURNAL entry." This makes the substitution legible rather than an unstated workaround.
- **`bin/next-fid.py` helper.** One-liner that greps TODO.md for the highest `F\d+` and prints the next ID. Saves ~20s per new-item allocation. Tag as `[easy] [infra]`.
- **Tag Live-Browser-required items as `[needs-mcp]`.** Items like F210, F212, and any future "browser-verify the inline confirm" task should carry an explicit marker so the routine builder doesn't pick them and bounce off the MCP gap. Skip-list in NEXT_RUN.md works as a manual fix, but a tag is more durable.
