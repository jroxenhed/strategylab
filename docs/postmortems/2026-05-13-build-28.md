## Friction points

- **Background agent + stop-hook tension**: the kieran-python reviewer returned synchronously (~58s) but the correctness reviewer ran in background mode (~458s — ~7.6 minutes). During that wait, the stop-hook fired ("There are uncommitted changes — please commit and push"), which would have implied "wrap up now." The right move was to keep waiting on the background agent (per the system prompt: "you will be automatically notified — do NOT sleep, poll, or proactively check"), but the stop-hook's prompt didn't compose cleanly with the wait protocol. I used the wait time to apply the safe_auto kieran fixes that didn't overlap with the deadline-helper logic the correctness reviewer was inspecting, which kept the session productive but introduces a small risk: if correctness had returned a finding on the constant naming or comment policy, those edits would have raced. They didn't (correctness stayed in its lane), but the pattern is fragile.
- **Manual fix disqualifies round-2 skip license**: the protocol's strict reading is "if ANY fix is autofix_class != safe_auto, re-dispatch round 2." For a test-timing constant bump (10ms → 50ms) that doesn't change production code, this feels like an over-application of the rule. Re-reading the rule, it's gated on "every applied finding was safe_auto" — and the round-2 cost was 21s, so the cost wasn't material. But a heuristic carve-out ("test-only timing constants don't count") would have saved the round trip. Not advocating for the carve-out today — the rule's bias toward safety is right — but noting the friction.
- **TODO line-number drift**: edited F127's bullet by string match on the unique long description, not by line number. That worked. The instruction set already warns about line-number drift; flagging that my muscle memory still wanted to grep for the line number first, which would have been wrong since I'd just regenerated the file via `sync-todo-index.py`.
- **`backend/venv/` missing for the 7th time (F97)**: 8-10 minutes of every backend-touching run go to `pip install fastapi pydantic numpy pandas httpx yfinance python-dotenv pytest pytest-asyncio alpaca-py tzdata`. The dependencies are stable across runs; pre-baking them in the routine container would compound across every future run. F97 has now been open for 7 builds without movement.

## Recommendations

- **Background-agent-vs-stop-hook protocol clarification**: add a note in `docs/overnight-builder-prompt-patch.md` §4 fix loop that explicitly says "while waiting for a background reviewer, you MAY apply safe_auto fixes from already-returned reviewers in lanes that don't overlap with the pending reviewer's focus. The stop-hook is informational; the canonical 'work is done' signal is the §5 final-verify pass." This codifies what I did, makes it deliberate not accidental, and gives the next builder explicit permission.
- **Heuristic carve-out for round-2 skip license**: amend §4 fix loop step 3 to add: "When the only non-`safe_auto` finding is a test-only timing constant or fixture data tweak (no production-code semantic change), the skip license still applies." Conservatively scoped; doesn't open the door to production-code manual fixes skipping review.
- **F97 priority bump**: this is the 7th overnight build that has had to runtime-install ~12 packages totaling several minutes. Cumulative cost across N builds dwarfs the one-time cost of pre-baking `backend/venv/` into the routine container image. File F97 as `[next]` in the next overnight builder run; currently it's not tagged and has been sitting in F · Testing and Infra.
- **F127 follow-ups well-decomposed**: the three follow-on items (F199 middleware option (c), F200 per-symbol watchdog, F201 first-symbol contract) are exactly the residual risks the correctness reviewer flagged plus the architectural deferral that F127 itself anticipated. No item is speculative; each maps to a concrete reviewer finding or design-doc reference. This is the kind of decomposition the §5 step 2.2 rule was written to produce.
